# 抖音直播联调笔记

## 当前结论

- 现阶段不要把抖音直播采集理解成“只靠一个 HTTP 接口就能全部搞定”。
- 我们本机已经确认页面首屏会稳定打出 `webcast/setting`、`webcast/user/me`、`webcast/web/similar_room_by_anchor`、多条 IM 相关请求，以及 `frontier-pc.douyin.com` / `frontier-im.douyin.com` 的 WebSocket 连接。
- 我们本机也已经确认页面上下文存在 `window.byted_acrawler.frontierSign`，并且页面内请求会自动补 `msToken` 和 `a_bogus`。
- 但首屏阶段没有看到 `/webcast/room/web/enter/` 被页面真实触发，所以它更像是“带前置条件的房间进入接口”，不能直接当成首个稳定入口。

## GitHub 经验

### 1. 直播流和弹幕通常会拆成两条链

- GitHub 上的 `wbt5/real-url` 明确把“直播源获取”和“弹幕获取”当成独立能力处理，说明实际工程里经常不是一个接口同时解决这两件事。
- 这对我们很重要：直播间基础状态、播放流、弹幕明细、评论/IM 很可能分别来自页面状态、HTTP 请求和 WebSocket。

### 2. UA、XBogus、ab 算法是一条链

- `Johnserf-Seed/f2` 的更新说明里明确强调：`XBogus` 支持自定义 `UA`，但 `UA` 必须规范；同时还开源了 `ab` 算法，并新增了直播弹幕转发功能。
- 这和我们本机观察是一致的：页面请求会把 `browser_name`、`browser_version`、`platform`、`webid`、`msToken`、`a_bogus` 一起带上，所以后面做真实 provider 时，签名参数、UA、Cookie 不能随便拆开替换。

### 3. Cookie 不是只要一个字段就够

- `B1gM8c/X-Bogus` 的 README 也提到，请求除了签名外，还需要携带 `msToken` 等 Cookie，上层业务请求不能只拼 URL。
- 这也和我们当前实测吻合：同样的 `room/web/enter`，就算在页面内拿 `frontierSign` 直接调用，如果参数链不满足，仍然会返回业务错误码。

## 对当前项目的直接影响

### 第一优先级

- 先把“页面真实发出的 HTTP/WSS 请求”抓稳定，而不是先把 `room/web/enter` 硬写进调度器。
- 目前更适合优先围绕这些已实测存在的路径推进：
  - `webcast/setting`
  - `webcast/user/me`
  - `webcast/web/similar_room_by_anchor`
  - `aweme/v1/web/im/...`
  - `frontier-pc.douyin.com/ws/v2`
  - `frontier-im.douyin.com/ws/v2`

### 第二优先级

- 继续定位 WebSocket 的消息格式和订阅参数。
- 如果弹幕、在线人数、互动计数最终主要走 WSS，那么后续稳定方案应当是：
  - 浏览器登录只负责获取和续期登录态
  - 业务进程复用 Cookie 和房间参数建立 WSS
  - 原始帧和解码后的消息同时落盘

### 第三优先级

- 对 `/webcast/room/web/enter/` 保持跟进，但先作为“候选接口”看待，不把它当唯一突破口。

- 然后再决定是先解 `frontier-im` 还是先解 `frontier-pc`。
