# 运行说明

## 1. Windows 本地启动

### 1.1 安装依赖

```bash
copy .env.example .env
py -3 -m pip install -e .
py -3 -m playwright install chromium
```

建议在 `.env` 里至少设置这几项：

```env
AUTO_CREATE_SCHEMA=true
DATABASE_URL=sqlite:///./runtime/app.db
SCHEDULER_ENABLED=false
DOUYIN_LIVE_PROVIDER=stub
APP_HOST=127.0.0.1
APP_PORT=8000
```

### 1.2 启动 API

```bash
py -3 -m uvicorn app.main:app --host 127.0.0.1 --port 8000 --app-dir src
```

### 1.3 初始化联调数据

```bash
curl -X POST http://127.0.0.1:8000/seed/demo
```

### 1.4 扫码导出抖音登录态

方式一，直接运行 Python CLI：

```bash
py -3 -m app.cli.export_douyin_storage_state --account-id douyin_demo --operator admin
```

方式二，使用 PowerShell 包装脚本：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\export_douyin_storage_state.ps1 -AccountId douyin_demo -Operator admin
```

执行后会自动打开 Chromium。你在浏览器里完成抖音扫码登录，然后回到终端按一次 Enter，系统会：

- 保存 `storage_state` 到 `BROWSER_STATE_DIR`
- 更新 `browser_login_state`
- 记录最新 `cookie_hash`

### 1.5 检查登录态

```bash
py -3 -m app.cli.inspect_login_state --account-id douyin_demo
```

或：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\inspect_douyin_login_state.ps1 -AccountId douyin_demo
```

### 1.6 切换到真实抖音联调模式

当登录态已经保存后，把 `.env` 调整为：

```env
DOUYIN_LIVE_PROVIDER=browser
DOUYIN_BROWSER_HEADLESS=true
SCHEDULER_ENABLED=true
```

然后重启 API，即可开始接真实 browser provider 的本地联调和定时扫描。

### 1.7 抓取真实房间的网络请求样本

当需要继续定位在线人数、弹幕接口或 WebSocket 时，可以直接开追踪脚本：

```bash
py -3 -m app.cli.trace_douyin_live_requests --account-id douyin_demo --room-url https://live.douyin.com/7044145585217083655
```

或：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\trace_douyin_live_requests.ps1 -AccountId douyin_demo -RoomUrl https://live.douyin.com/7044145585217083655
```

它会打开带登录态的 Chromium，并把页面里的 `fetch/xhr/websocket` 请求写到 `data/raw/douyin/request-trace/*.jsonl`。

抓完之后可以直接分析：

```bash
py -3 -m app.cli.analyze_douyin_trace --input data/raw/douyin/request-trace/douyin_demo-20260402T051008Z.jsonl
```

它会输出：

- 事件和资源类型统计
- 高频 `xhr/fetch` 路径
- WebSocket 地址
- WebSocket 帧预览
- 包含 `webcast`、`room`、`comment`、`im`、`danmaku` 的候选请求

如果要把一个直播间的已知候选接口一次性探一遍，可以直接跑：

```bash
py -3 -m app.cli.probe_douyin_live_room --room-id 7044145585217083655 --account-id douyin_demo
```

它会把这些结果一起落到 `data/raw/douyin/probe/*.json`：

- 房间页解析结果
- `webcast/setting`
- `webcast/user/me`
- `webcast/web/similar_room_by_anchor`

目前本地实测里：

- `webcast/setting` 可用
- `webcast/user/me` 可用
- `similar_room_by_anchor` 当前返回 `status_code=10011`

如果要在浏览器页面上下文里复用 Douyin 前端自己的 `frontierSign` 做带签名请求，可以直接跑：

```bash
py -3 -m app.cli.probe_douyin_signed_api --account-id douyin_demo --room-id 7044145585217083655 --preset room-web-enter
```

或：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\probe_douyin_signed_api.ps1 -AccountId douyin_demo -RoomId 7044145585217083655 -Preset room-web-enter
```

它会把这些信息一起落到 `data/raw/douyin/browser-probe/*.json`：
- 当前页面运行上下文
- `window.byted_acrawler.frontierSign` 返回的签名头
- 页面内 `fetch` 的真实响应

如果要继续观察页面运行时在加载阶段自己触发了哪些 `fetch/xhr/websocket`，以及有没有真正调用暴露在页面上的 `frontierSign`，可以直接跑：

```bash
py -3 -m app.cli.trace_douyin_page_runtime --account-id douyin_demo --room-id 7044145585217083655 --wait-seconds 15
```

或：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\trace_douyin_page_runtime.ps1 -AccountId douyin_demo -RoomId 7044145585217083655 -WaitSeconds 15
```

它会把结果落到 `data/raw/douyin/page-runtime-trace/*.json`，并额外给出摘要：
- 运行时事件计数
- 页面实际命中的关键路径
- 是否看到了 `/webcast/room/web/enter/`
- 是否看到了 `frontierSign` 调用

如果要把已经抓到的 trace 里的 `frontier-pc` / `frontier-im` WebSocket 握手参数单独拆出来，可以直接跑：

```bash
py -3 -m app.cli.extract_douyin_frontier_ws --input data/raw/douyin/request-trace/douyin_demo-20260402T053837Z.jsonl
```

或：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\extract_douyin_frontier_ws.ps1 -Input data/raw/douyin/request-trace/douyin_demo-20260402T053837Z.jsonl
```

它会输出并落盘这些内容：
- `frontier-pc` / `frontier-im` 的完整握手 URL
- query 参数拆解结果
- `aid`、`device_id`、`access_key`、`conn_tag`、`fpid`、`version_code`
- 已关联到的首批 WebSocket frame 预览

如果要继续检查 `frontier` frame 的二进制内容，可以直接跑：

```bash
py -3 -m app.cli.inspect_douyin_frontier_frames --input data/raw/douyin/request-trace/douyin_demo-20260402T053837Z.frontier.json
```

或：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\inspect_douyin_frontier_frames.ps1 -Input data/raw/douyin/request-trace/douyin_demo-20260402T053837Z.frontier.json
```

它会给出：
- 编码猜测
- 原始字节长度
- hex / ASCII 预览
- 轻量 protobuf 风格字段拆解

如果已经拿到一份包含 `cdp_websocket_frame_received` 的真实 trace，需要验证“解码 → comment 表入库”闭环，可以直接回放：

```bash
py -3 -m app.cli.import_douyin_trace_comments \
  --input data/raw/douyin/request-trace/douyin_demo-20260402T171003Z.jsonl \
  --session-id 46
```

说明：
- `--input` 传一份历史真实 WSS trace；
- `--session-id` 传目标直播场次 ID；
- 工具会先调用 `websocket_decoder` 解码，再把 `chat/member/gift/like/social` 这几类消息写入 `douyin_live_comment`；
- 可用于协议回归测试、数据库写入验证、以及在线抓帧受验证码影响时的离线回放联调。

### 2026-04-03 已验证样例

已用真实在播房间 `https://live.douyin.com/572695244335` 完成一次闭环验证：

1. `trace_douyin_live_requests` 成功抓到：
   - `cdp_websocket_created` = 1
   - `cdp_websocket_frame_received` = 32

   - `Total frames = 64`
   - `Total messages = 59`
   - `member = 52`
   - `chat = 1`
3. `import_douyin_trace_comments --session-id 46` 成功入库：
   - `inserted = 53`
   - comment 表中 `session_id=46` 的记录数 = `53`
4. 当前 `WebcastChatMessage` 已增加 fallback，可解出样例聊天内容：
   - `杨***: 敲出来以后，可以把背后弄薄一点吗`

### 2026-04-03 稳定性复核

本轮未继续扩大功能面，只做闭环核对与稳定性验证：

1. 对历史真实 trace `data/raw/douyin/request-trace/douyin_demo-20260402T171003Z.jsonl` 再次执行：
   - `decode_websocket`
   - `import_douyin_trace_comments --session-id 46`
2. 结果确认：
   - 解码结果仍为 `Total frames = 64`、`Total messages = 59`
   - 回放导入结果为 `inserted=0 duplicate=53 repaired=0 skipped=6 failed=0`
   - 说明 comment 入库已具备**幂等性**，重复回放不会重复写入
3. 数据库复核：
   - `session_id=46` 的 comment 总数仍为 `53`
   - 类型分布：`chat=1`、`member=52`
4. 新抓的 `data/raw/douyin/request-trace/douyin_demo-20260403T030626Z.jsonl` 未拿到 WebSocket 帧；首屏 HTML 明确是“验证码中间页”，说明当前实时抓帧的主要不稳定因素仍是**风控/验证码挑战**，而不是 decoder / comment 入库链路本身。

注意：
- 运行脚本时建议始终以**仓库根目录**作为工作目录；
- 如果当前环境没有执行 `pip install -e .`，而是直接用仓库内 `venv/bin/python` 调 CLI，请显式加上 `PYTHONPATH=src`（例如：`PYTHONPATH=src venv/bin/python -m app.cli.decode_websocket ...`），否则会报 `ModuleNotFoundError: No module named 'app'`；
- 项目现在会把 `.env`、`RAW_DATA_DIR`、`BROWSER_STATE_DIR` 都解析到仓库根，避免误写到 `src/data/raw` 一类路径；
- 若测试房间已下播，抓不到 WebSocket 帧是正常现象，应先换一个当前真实在播房间再验证；
- 若页面首屏返回“验证码中间页”，需先处理 challenge / 登录态风控问题，否则实时 trace 可能只有验证码请求，没有直播 WSS。
- Browser provider 现已支持 `DOUYIN_CHALLENGE_RETRY_SECONDS`（默认 900 秒）：登录态被标记为 `challenge` 后，不会永久跳过 authenticated sidecar，而是在冷却窗口后自动再试一次；`/monitor/douyin/live/sidecar-stats` 的 `sidecar_errors` 会显示 `retry-after` 剩余秒数，便于判断何时会自动重试。

## 2. 样本联调

推荐先跑一次样本链路，确认数据库、查询接口、JSONL 归档都正常：

1. `POST /seed/demo`
2. 用 `data/samples/douyin_live_status_sample.json` 调 `POST /douyin/ingest/status-sample`
3. 用 `data/samples/douyin_live_comment_sample.json` 调 `POST /douyin/ingest/comment-sample`
4. 用直播查询接口验证场次、快照、弹幕是否入库

## 6. Douyin Provider 说明

当前支持两个 provider 值：

- `stub`
- `http`

说明：

- `stub`：本地开发占位实现，不访问真实抖音接口
- `http`：已支持复用登录态抓真实房间页，并解析 `live_status`、`web_rid` 等页面级状态；在线人数、弹幕和更细指标继续通过请求追踪补齐
00`（5 分钟）以匹配多房间串行扫描的实际耗时
