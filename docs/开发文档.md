# 抖音小红书数据监测开发文档

## 1. 项目目标

建设一个可持续运行的数据监测系统，覆盖抖音与小红书两大平台，支持多账号、多直播间、多内容对象的持续采集、状态追踪、历史留存、异常告警与基础分析。

系统目标包括：

- 统一管理抖音直播间、小红书账号、抖音账号等监测对象
- 定时采集各类对象的实时与增量数据
- 支持 30 秒级轮询的直播监测
- 支持粉丝变化、发文变化、互动变化等趋势追踪
- 支持后续扩展分析看板、消息通知与报表导出

## 2. 需求范围

### 2.1 抖音部分

#### 2.1.1 直播间持续监测

目标：持续监测多个抖音直播间的开播状态，并在直播期间以 30 秒轮询的方式抓取直播基础数据与互动数据。

核心需求：

- 支持配置多个直播间监测对象
- 持续检测直播间是否开播、下播
- 开播后自动进入高频采集
- 轮询间隔默认 30 秒
- 支持直播结束后自动停止高频采集并记录直播场次摘要

- 是否为管理员 / 主播 / 粉丝团成员（如可获取）

衍生能力：

- 自动识别开播与下播事件
- 生成单场直播汇总数据
- 按分钟/小时汇总互动趋势
- 支持敏感词、关键词、竞品词弹幕监测

## 3. 系统设计

### 3.0 技术实现方案定稿

结合后续部署环境为 Linux、数据量会持续增长、直播监测存在 30 秒高频轮询、弹幕与评论明细需要尽可能详细留存这几个前提，本项目采用“关系型数据库 + 原始文本归档”的混合方案。

最终建议方案如下：

- 应用语言：Python 3.9+，生产环境建议 3.11+
- Web/API：FastAPI
- 调度：APScheduler
- 数据库：PostgreSQL
- 缓存与轻量队列：Redis
- 原始数据归档：JSONL 文本文件
- 浏览器自动化：Playwright 优先，Selenium 作为备选
- 部署方式：Docker Compose 部署到 Ubuntu
- 日志：结构化日志，落本地文件并按天切分

决策原因：

- PostgreSQL 适合做账号、视频、笔记、评论、粉丝变化、直播快照等结构化查询
- JSONL 适合保存直播弹幕、原始接口返回、异常样本，便于回放和追溯
- APScheduler 足够覆盖当前定时监测场景，复杂度低于 Celery，适合第一阶段快速上线
- Redis 可用于分布式锁、去重键、任务状态缓存、短期游标缓存
- 浏览器自动化是抖音、小红书这类平台获取登录态和处理风控的重要补充
- Docker Compose 适合单机或单台云服务器 Ubuntu 部署，后续也容易迁移到容器平台

结论：

- 核心业务数据存 PostgreSQL
- 原始接口响应与超详细明细同时写入 JSONL 归档
- 不采用“纯文本存储”作为主方案，因为后续趋势统计、筛选、联表分析会非常麻烦
- 抖音登录态获取采用“浏览器模拟登录 + Cookie/StorageState 复用”的方案
- 浏览器自动化优先使用 Playwright，不把 Selenium 作为默认主方案

### 3.1 模块划分

建议按以下模块拆分：

- `config`：平台账号、直播间、轮询频率、告警规则等配置管理
- `browser`：浏览器自动化、登录态管理、Cookie 刷新
- `collector`：平台采集逻辑
- `scheduler`：定时调度与任务分发
- `parser`：原始返回数据标准化
- `storage`：数据库写入与文件落盘
- `service`：业务聚合逻辑，例如直播状态判断、粉丝增量计算
- `alert`：异常告警、开播提醒、数据变化提醒
- `api`：后端接口，供管理页面或看板调用
- `dashboard`：可视化页面

按平台进一步拆分：

- `collector/douyin/live`
- `collector/douyin/account`
- `collector/xiaohongshu/account`
- `collector/xiaohongshu/note`

### 3.2 任务类型

建议定义以下任务类型：

- 登录态刷新任务
- 直播间开播状态检测任务
- 直播高频轮询任务
- 抖音账号主页采集任务
- 抖音视频列表采集任务
- 小红书账号主页采集任务
- 小红书笔记列表采集任务
- 小红书笔记评论采集任务
- 数据汇总与告警任务

### 3.3 调度策略

建议调度方式：

- 登录态有效性检查：5 分钟到 15 分钟一次
- 登录态自动刷新：按失效风险和人工触发执行
- 直播开播检测：15 秒到 60 秒一次，可配置
- 直播数据轮询：固定 30 秒一次
- 抖音账号粉丝监测：5 分钟到 30 分钟一次
- 抖音视频发布监测：5 分钟到 30 分钟一次
- 小红书账号粉丝监测：10 分钟到 60 分钟一次
- 小红书笔记互动监测：5 分钟到 30 分钟一次
- 小红书评论拉取：对新笔记高频，对老笔记降频

建议加入以下机制：

- 错峰调度，避免所有任务同一秒触发
- 失败重试机制
- 指数退避机制
- 任务幂等控制
- 任务超时控制
- 反爬风控保护
- 登录态失效自动熔断与人工接管

## 4. 数据存储设计

### 4.1 推荐技术选型

本项目最终采用以下方案：

- 应用语言：Python 3.9+，生产环境建议 3.11+
- Web/API：FastAPI
- 调度：APScheduler
- 数据库：PostgreSQL
- 缓存与队列：Redis
- 原始数据归档：本地 JSONL 文件，必要时可扩展对象存储
- 浏览器自动化：Playwright 优先，Selenium 备选
- 日志：结构化日志 + 文件归档

不建议作为正式方案的选项：

- SQLite：不适合后续高频写入与并发采集
- 纯文本为主的数据存储：不利于查询、告警、汇总与管理后台建设

### 4.2 核心数据表建议

建议至少包含以下表：

- `platform_account`：监测账号配置
- `douyin_live_room`：抖音直播间配置
- `douyin_live_session`：直播场次
- `douyin_live_snapshot`：直播轮询快照
- `douyin_live_comment`：直播弹幕明细
- `douyin_account_snapshot`：抖音账号快照
- `douyin_video`：抖音视频主表
- `douyin_video_snapshot`：抖音视频指标快照
- `xiaohongshu_account_snapshot`：小红书账号快照
- `xiaohongshu_note`：小红书笔记主表
- `xiaohongshu_note_snapshot`：小红书笔记指标快照
- `xiaohongshu_note_comment`：小红书评论明细
- `task_log`：任务执行日志
- `alert_event`：告警事件

### 4.3 监测账号配置字段设计

账号属性需要支持业务分类与管理属性，建议统一收敛到 `platform_account` 主表中。

建议字段如下：

- `id`：内部主键
- `account_no`：内部编号，便于人工管理和报表引用
- `platform`：平台类型，取值如 `douyin`、`xiaohongshu`
- `account_id`：平台账号唯一标识
- `account_handle`：抖音号 / 小红书号 / 可读账号名
- `nickname`：昵称
- `account_type`：账号分类，例如品牌号、达人号、门店号、竞品号、观察号
- `is_competitor`：是否竞品
- `department`：所属部门
- `owner`：内部负责人
- `priority`：监测优先级
- `status`：是否启用监测
- `homepage_url`：主页链接
- `live_room_url`：直播间链接，适用于抖音直播账号
- `discover_source`：线索来源，例如市场部、销售部、手工录入
- `tags`：标签数组，例如品类、品牌、业务线、区域
- `notes`：备注
- `created_at`：创建时间
- `updated_at`：更新时间

补充说明：

- `account_no` 建议作为业务编号长期保留，不依赖平台 ID
- `is_competitor` 与 `department` 是一等字段，不只放备注，便于过滤和统计
- `homepage_url`、`live_room_url` 必须保留，方便人工复核与补采
- `tags` 建议使用 PostgreSQL `text[]` 或 `jsonb`

### 4.4 快照与明细分层

建议按以下方式存储：

- 主表：保存对象稳定属性，例如账号 ID、笔记 ID、视频 ID
- 快照表：保存每次采集的指标数值，例如粉丝数、点赞数、评论数
- 明细表：保存互动明细，例如弹幕、评论
- 原始表或原始文件：保存平台返回的原始 JSON

这样有利于：

- 做历史趋势分析
- 减少字段污染
- 保证追溯能力
- 便于后续重放解析逻辑

### 4.5 弹幕明细存储策略

弹幕明细要求“尽可能详细”，因此采用“双写策略”：

- 结构化字段写入 PostgreSQL `douyin_live_comment`
- 完整原始消息体写入 JSONL 归档文件

原因：

- PostgreSQL 便于按用户、关键词、直播场次、时间范围快速检索
- JSONL 可无损保留所有原始字段，后续即使解析规则升级也能重新回放

建议的数据库字段如下：

- `id`：内部主键
- `platform`：平台标识
- `session_id`：直播场次 ID
- `room_id`：直播间 ID
- `message_id`：弹幕消息唯一标识
- `message_type`：消息类型，例如普通弹幕、进场、点赞、礼物、关注
- `event_time`：事件发生时间
- `fetch_time`：采集时间
- `sequence_no`：接口返回顺序号或本地顺序号
- `user_id`：发送用户 ID
- `sec_user_id`：平台加密用户 ID，如可获取
- `nickname`：用户昵称
- `display_name`：展示名
- `avatar_url`：头像链接
- `gender`：性别，如可获取
- `city`：地区，如可获取
- `province`：省份，如可获取
- `country`：国家，如可获取
- `follower_count`：用户粉丝数，如可获取
- `fan_level`：粉丝团等级，如可获取
- `user_level`：用户等级，如可获取
- `is_anchor`：是否主播本人
- `is_admin`：是否管理员
- `is_room_manager`：是否场控
- `is_fans_group_member`：是否粉丝团成员
- `is_new_fan`：是否新进粉
- `content`：弹幕正文
- `content_plain`：去格式化纯文本
- `emoji_text`：表情文本映射
- `mentioned_users`：@ 用户列表
- `extra_badges`：徽章信息
- `device_info`：终端信息，如可获取
- `ip_location`：IP 属地，如可获取
- `risk_flags`：风险标记
- `raw_json`：当前记录对应的原始 JSON 子对象
- `raw_file_path`：原始 JSONL 文件路径
- `raw_line_no`：原始 JSONL 行号
- `created_at`：入库时间

JSONL 归档建议：

- 按平台 / 日期 / 房间 / 场次分目录
- 每行一条原始消息
- 文件名包含开始时间与直播场次 ID
- 原始响应尽量整包保存，不只保留解析后的子字段

建议目录格式：

```text
data/raw/douyin/live_comments/2026-04-02/{room_id}/{session_id}.jsonl
```

建议额外保留：

- 请求参数
- 响应头
- 分页游标
- 抓取节点标识
- 错误响应样本

## 5. 采集流程设计

### 5.1 抖音直播监测流程

1. 定时轮询直播间状态
2. 若未开播，记录状态并等待下一次检测
3. 若接口需要登录态，则优先使用已缓存的浏览器登录态
4. 若检测到开播，创建直播场次记录
5. 启动 30 秒高频采集任务
6. 每 30 秒抓取一次直播基础数据
7. 每次采集同步拉取新增弹幕明细
8. 直播结束时关闭场次，并生成汇总数据

### 5.2 登录态与浏览器自动化流程

1. 通过 Ubuntu 环境中的 Chromium 浏览器执行首次人工扫码登录
2. 登录成功后保存 Cookie、Local Storage、Session Storage 或 Playwright `storage_state`
3. 采集任务优先复用现有登录态访问接口
4. 若发现接口返回未登录、风控校验、验证码等信号，则记录风险状态
5. 系统暂停高风险账号采集并触发登录态刷新任务
6. 刷新成功后恢复该账号或该浏览器上下文的采集任务

推荐实现方式：

- 首选 Playwright
- 需要兼容既有脚本时可保留 Selenium 适配层
- 不建议把每次采集都放在浏览器里执行，成本太高
- 推荐“浏览器负责登录态获取，请求层负责正式采集”的混合模式

原因：

- 浏览器直接采集直播和评论，资源占用会很重
- 30 秒轮询多个直播间时，纯 Selenium 会明显增加机器压力
- 用浏览器拿 Cookie，再用 `httpx`/`requests` 访问接口，更适合高频任务

### 5.3 抖音账号监测流程

1. 定时拉取账号主页
2. 写入账号快照
3. 计算粉丝变化量
4. 拉取视频列表
5. 对比新旧视频 ID，识别新增、删除、隐藏变化
6. 对新增或重点视频补拉详情与互动数据

### 5.4 小红书监测流程

1. 定时拉取账号主页并记录粉丝快照
2. 拉取笔记列表并更新笔记主表
3. 对新增笔记拉取详情
4. 对监测范围内笔记采集点赞、评论、收藏等指标
5. 对重点笔记拉取评论列表并做增量入库
6. 根据规则触发告警或报表更新

## 6. 接口与反爬考虑

本项目在实施时需要重点关注平台接口稳定性与合规性。

关键注意点：

- 不同平台可能存在 Web 接口、移动端接口、H5 接口差异
- 登录态、Cookie、签名算法、设备指纹可能影响采集成功率
- 高频采集尤其是 30 秒轮询直播数据，必须控制请求速率
- 需设计代理、重试、限流、熔断机制
- 需区分“监测对象频率”和“单接口频率”两个层次
- 浏览器自动化流程需要应对验证码、设备校验、短信校验、扫码登录

建议的工程措施：

- 请求层统一封装
- 登录态统一管理
- 失败码统一分类
- 对接口版本变化保留兼容层
- 关键原始响应做样本归档
- 为浏览器登录态单独建立状态表和过期检测
- 浏览器上下文与采集任务解耦

### 6.1 浏览器自动化选型

建议主方案：

Selenium 的定位：

## 11. 风险与注意事项

- 平台接口可能随时变化
- 登录态失效会导致采集失败
- 高频轮询可能触发风控
- 粉丝数、在线人数等字段可能存在平台展示口径差异
- 评论 / 弹幕可能存在去重与增量抓取难点
- 小红书和抖音不同终端返回结构可能差异较大

因此建议从一开始就保留：

- 原始响应归档
- 统一错误日志
- 可回放的解析样本
- 可配置的频率与限流策略

## 12. Linux 部署建议

建议默认部署环境：

- Ubuntu 22.04 LTS 或兼容的 Linux 发行版
- Python 3.11
- PostgreSQL 15+
- Redis 7+
- Chromium
- Playwright 运行时依赖
- Docker Engine + Docker Compose Plugin

建议服务拆分：

- `app`：采集主服务，负责调度与采集
- `api`：查询接口服务，可与 `app` 合并部署
- `postgres`：主数据库
- `redis`：缓存与锁
- `nginx`：反向代理，可选

建议挂载目录：

- `/opt/monitoring/app`：应用代码
- `/opt/monitoring/data`：原始 JSONL 数据
- `/opt/monitoring/logs`：日志
- `/opt/monitoring/config`：环境配置
- `/opt/monitoring/browser`：浏览器登录态与存储状态

部署原则：

- 容器内运行，宿主机挂载数据目录
- 数据库与原始文件均持久化
- 时区统一为 `Asia/Shanghai`
- 所有服务通过环境变量管理配置
- 浏览器登录态目录单独挂载并严格控制权限

建议补充的部署脚本：

- `deploy/ubuntu/install_dependencies.sh`：在 Ubuntu 上安装 Python、Redis、PostgreSQL、Chromium、Docker、Compose
- `deploy/ubuntu/bootstrap_server.sh`：初始化部署目录、用户、权限和环境文件模板
